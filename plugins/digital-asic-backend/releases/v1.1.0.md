## Migrating an Existing Project

Run the following from your project's `fc/` directory. The commands are idempotent â€” safe to run even if some renames have already been applied.

```bash
cd $PROJECT_PATH/fc

# 1. Rename runners/ â†’ flows/
[ -d scripts/runners ] && mv scripts/runners scripts/flows

# 2. Rename steps/ â†’ stages/
[ -d scripts/steps ] && mv scripts/steps scripts/stages

# 3. Rename design_setup files
[ -f scripts/stages/design_setup.tcl ] && mv scripts/stages/design_setup.tcl scripts/stages/init.tcl
[ -f scripts/flows/run_design_setup.tcl ] && mv scripts/flows/run_design_setup.tcl scripts/flows/run_init.tcl

# 4. Update source paths inside all TCL files
find scripts -name "*.tcl" -exec sed -i \
    -e 's|scripts/steps/|scripts/stages/|g' \
    -e 's|\.\./steps/|\.\./stages/|g' \
    -e 's|stages/design_setup\.tcl|stages/init.tcl|g' \
    -e 's|scripts/runners/|scripts/flows/|g' \
    -e 's|\.\./runners/|\.\./flows/|g' \
    {} +

# 5. Update Makefile
sed -i \
    -e 's|runners/|flows/|g' \
    -e 's|run_design_setup|run_init|g' \
    -e 's|^\.PHONY:.*design_setup|& |;s|design_setup\b|init|g' \
    -e 's|^design_setup:|init:|' \
    Makefile
```

> **Note:** `constraints/antenna_rules.tcl` in existing projects is unaffected â€” your PDK-specific content is already in place. The placeholder only applies to projects created fresh from the v1.1.0 template.

## New Workflow: PDK Discovery

A new skill `/discover-pdk` replaces the inline PDK search that was embedded in `/new-project`.

**For new projects**, run this once per PDK:
```
/discover-pdk $PDK_PATH
```
This writes `$PDK_PATH/pdk.md` â€” a structured catalog of all PDK file paths. `/new-project` reads it directly on subsequent runs, skipping the slow `find` scans.

**For existing projects**, no action is required. You only need `pdk.md` if you plan to scaffold additional new projects from the same PDK.

---

## Detail: `runners/` â†’ `flows/`, `steps/` â†’ `stages/`

`runners/` implied generic script execution. The files actually define *which stages to invoke*, making `flows/` the more accurate name. `stages/` replaces the generic `steps/` with the standard PnR term for each implementation phase.

`stages/design_setup.tcl` was renamed `stages/init.tcl` because `design_setup` overlapped conceptually with the project-level `setup.tcl` in the parent directory. `init.tcl` unambiguously describes what the stage does: initialize the design database (elaborate RTL, load TLU+, apply constraints).

## Detail: `discover-pdk` + `pdk.md`

`/new-project` previously ran a full PDK `find` scan on every invocation. The new approach separates concerns:

- **`/discover-pdk`** â€” one-time scan, confirms paths with user, writes `$PDK_PATH/pdk.md`
- **`/new-project`** â€” reads `pdk.md`, no filesystem scan

`pdk.md` is a markdown table (`| Variable | Value |`) organized into sections. It lives in the PDK directory and is reusable across all projects on that PDK. Any variable not found during discovery is recorded as `MISSING`; `/new-project` warns on these before proceeding.
