## Summary

| Area | Change |
|------|--------|

## `/discover-pdk` fixes

### NDM libraries are directories, not files

A `.ndm` library such as `stdcell.ndm` is a directory containing internal files (e.g. `reflib.ndm`); previous discovery would `find -type f -name "*.ndm"` and pick a file from inside the library. Searches now use `find -type d -name "*.ndm"` so the library root is selected.

### Portable `pdk.md`

All paths written into `pdk.md` are now relative to the directory containing `pdk.md` (i.e. `$PDK_PATH`). This keeps the summary file valid if the PDK is later remounted at a different absolute location.

`TARGET_LIBRARY` and `SYMBOL_LIBRARY` remain basenames only (no path component).

### `/new-project` consumption

`/new-project` reads each value from `pdk.md` and prepends `$PDK_PATH` to recover the absolute path used in template substitution. `LIBRARY_PATH` is always set to `$PDK_PATH` itself (the argument the user provided), independent of any value inside `pdk.md`.

If you have a `pdk.md` written by v1.1.0, **regenerate it** with `/discover-pdk $PDK_PATH` so the paths are stored in the new relative form.

## RTL read-in via VCS filelist

### What changed

`stages/init.tcl` previously used `analyze -autoread`, which is deprecated and gave inconsistent results on multi-file SystemVerilog projects. It now reads RTL through a VCS-style filelist:

```tcl
analyze -vcs "-sverilog -f ${PROJECT_PATH}/hdl/filelist.f"
```

If `hdl/filelist.f` is missing on a fresh project, `init.tcl` writes an empty placeholder with usage comments and prints a warning, instead of letting `analyze` fail with a less informative error.

### Why `-f`, not `-F`

Both flags exist in VCS:

- `-f <file>` — paths in the file are relative to the **invocation directory** (in our flow, `fc/temp/`).
- `-F <file>` — paths are relative to the **filelist's own directory**.

We use `-f` per VCS convention. The placeholder auto-written by `init.tcl` and the template `hdl/filelist.f` both use absolute paths (with `$PROJECT_PATH` substituted at scaffold time) so users do not have to think about which directory paths resolve from.

### Filelist format

`hdl/filelist.f` accepts standard VCS directives:

```
// Absolute paths (recommended)
/path/to/proj/hdl/top.sv
/path/to/proj/hdl/sub/alu.sv
+incdir+/path/to/proj/hdl/include
+define+SYNTHESIS

// Relative paths (resolved from fc/temp/)
../../hdl/top.sv
+incdir+../../hdl/include

// Nested filelists
-f /path/to/other_filelist.f
```

### Action for users

- **New projects scaffolded under v1.1.1** — `hdl/filelist.f` is created from the template with absolute paths pre-filled. Add your `.sv`/`.v` lines and run `/run-fc`.
- **Projects scaffolded under v1.1.0** — replace `analyze -autoread ...` in `fc/scripts/stages/init.tcl` with the `analyze -vcs -f` line above, and create `hdl/filelist.f` (or let `init.tcl` create the placeholder on first run).
